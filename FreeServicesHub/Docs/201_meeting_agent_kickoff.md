## Discussion

**[CTO]:** Alright, here's what we're building. FreeServicesHub.Agent -- it's a background service that runs on remote machines and reports back to the FreeServicesHub web app via SignalR. Think of it like an Azure DevOps runner, but ours. The hub has a page where you can see all your agents, their status, heartbeats, logs -- everything in real time.

The interesting part is how the keys work. There are actually two layers of keys. The first is a **registration key** -- that comes from the CI/CD pipeline. When we deploy, Azure DevOps generates a short-lived key and passes it to the agent during installation. The agent uses that key to register with the hub. Once registered, the hub generates a **session key** -- like we do in FreeGLBA with the API client token -- and hands it back to the agent. From that point on, the agent uses the session key for all communication. The registration key expires.

**[Architect]:** Let me make sure I understand the trust chain. The CI/CD pipeline is the root of trust -- it knows the shared deployment key. The agent uses that to prove it's a legitimate install. Then the hub takes over and issues its own key. So we have three things: the deployment key, the registration key, and the session key?

**[CTO]:** Almost. Let me simplify. There are two keys:

1. **Registration key** -- Generated by the CI/CD pipeline, one per agent instance, per deployment. It's a one-time-use key. You use it to register, then it's burned. Can't register twice with the same key.
2. **API client token** -- Generated by the hub when an agent registers. This is the long-lived key the agent uses for heartbeats, log streaming, everything. The hub can revoke it at any time to disconnect an agent.

**[Security]:** One-time-use registration keys -- I like that. So during a deployment, if we're deploying to three servers, the pipeline generates three unique registration keys?

**[CTO]:** Exactly. Three servers, three keys. Each key is valid for one registration only. And they expire after 24 hours regardless, so even if one doesn't get used, it's not a lingering vulnerability.

**[Security]:** And the API client token the hub generates -- that follows the FreeGLBA pattern? SHA-256 hash stored in the database, plaintext given to the agent once during registration, never stored on the hub side?

**[CTO]:** Yes. The agent stores its token locally -- in its appsettings.json or wherever. The hub only stores the hash. If I want to disconnect an agent, I revoke the token from the hub UI. The agent's next heartbeat fails auth, and it knows it's been disconnected.

**[Backend]:** Walk me through the deployment lifecycle. What's the sequence of events when we push a new build?

**[CTO]:** Here's the full cycle:

1. CI/CD pipeline triggers -- either manually or on schedule. I have mine running at 2 AM daily.
2. Pipeline generates N registration keys (one per target server) and pushes them to the hub via API.
3. Pipeline sends a **shutdown command** to all agents through the hub. The hub broadcasts via SignalR: "shut down."
4. Agents receive the command, gracefully stop, and confirm shutdown.
5. Hub reports all agents are offline.
6. Pipeline shuts down the hub itself.
7. Pipeline deploys the new hub build and starts it.
8. Pipeline deploys the new agent build to each target server, writing the registration key into the agent's config.
9. Each agent starts up, uses its one-time registration key to register with the hub.
10. Hub validates the registration key, burns it, generates an API client token, returns it.
11. Agent stores the token and begins heartbeat loop.
12. Hub dashboard shows all agents online.

The whole thing takes a few minutes. And since it runs at 2 AM, nobody notices. If something breaks, the worst case is the agents are offline until the next deployment -- never more than 24 hours.

**[Architect]:** So the hub itself also gets redeployed in every cycle. That means the hub's database persists across deployments -- the API client tokens survive the restart because they're in the database, not in memory.

**[CTO]:** Right. The database is the source of truth. The hub restarts, reconnects to the same database, and all the token hashes are still there. When agents reconnect after deployment, they register fresh with new tokens anyway, so the old ones don't matter.

**[AgentDev]:** For the agent side -- the worker loop. Right now in FreeServices, the `SystemMonitorService` collects system snapshots and writes to a log file every N seconds. I'm replacing that with a SignalR heartbeat. But what data do we send in the heartbeat?

**[CTO]:** At minimum: "I'm alive, here's my agent ID, here's my timestamp." But we should also send the system snapshot data -- CPU, memory, disk -- like the existing service does. And any logs the agent has collected since the last heartbeat. Think of it as a combined health check and log drain.

**[AgentDev]:** And if the SignalR connection drops? The agent should reconnect automatically?

**[CTO]:** Yes. The agent should keep trying to reconnect with exponential backoff. If it can't reconnect after some reasonable period, it should fall back to logging locally -- like the current service does -- so we don't lose data. When the connection comes back, it can flush the buffered logs.

**[Quality]:** How do we test this? The FreeServices.TestMe project has a great pattern with 4 test modes. I'm thinking we need similar coverage:

1. **Console mode** -- Agent runs as a process, connects to a local hub, sends heartbeats.
2. **Platform service mode** -- Agent installed as a real OS service, registers with hub, verified via API.
3. **Registration flow** -- Test the one-time key lifecycle: generate, register, burn, reject reuse.
4. **Revocation flow** -- Register, get token, revoke from hub, verify agent gets disconnected.

**[CTO]:** That's exactly right. And we should be able to run all of that in CI.

**[Architect]:** One more question. The hub's SignalR page for monitoring agents -- is this a new page we're building, or are we extending the existing FreeCICD-style dashboard?

**[CTO]:** New page. But we should study how FreeCICD does their pipeline dashboard with progressive loading and the `PipelineMonitorService` background service. Same concept -- a background service that tracks connected agents, detects changes, and pushes updates to the dashboard via SignalR. But our domain is agents, not pipelines.

**[Security]:** Last question. The registration keys that the CI/CD pipeline generates -- where do they come from? Does the pipeline call a hub API endpoint to request them? Or does the hub generate them in advance?

**[CTO]:** The pipeline calls a hub API endpoint. Something like `POST /api/Data/GenerateRegistrationKeys` with a count parameter. The hub generates the keys, stores their hashes with a 24-hour expiry and a "one-time-use" flag, and returns the plaintext keys to the pipeline. The pipeline then injects each key into the corresponding agent's deployment.

**[Security]:** So the hub needs to be running when the pipeline starts -- before we shut everything down for deployment. That means the sequence is: generate keys while hub is still up, then shutdown, deploy, restart.

**[CTO]:** Exactly. The hub is the key authority. Everything flows through it.

## Decisions

1. **Two-key model**: One-time registration keys (from CI/CD) and revocable API client tokens (from hub).
2. **Registration keys are per-instance, per-deployment, one-time-use, 24-hour expiry.**
3. **API client tokens follow the FreeGLBA pattern**: SHA-256 hash in DB, plaintext to agent once.
4. **Hub can revoke any agent** by revoking its API client token.
5. **Deployment lifecycle**: Generate keys -> shutdown agents -> shutdown hub -> deploy hub -> deploy agents -> agents register -> hub issues tokens.
6. **Agent heartbeat** includes: alive signal, agent ID, timestamp, system snapshot (CPU/memory/disk), buffered logs.
7. **Agent reconnection**: Exponential backoff on SignalR disconnect, local log buffering as fallback.
8. **Nightly deployment** at 2 AM ensures max 24-hour downtime for any broken deploy.
